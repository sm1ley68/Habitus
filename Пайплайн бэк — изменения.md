# 🛠️ Для бэкенда: все изменения по карточке-«Досье»

> Самодостаточная сводка того, что бэку нужно реализовать после переработки паспорта объекта в формат **«Досье детектива»** (кинематографичная карточка + чат по объекту).
>
> Это выжимка из `Пайплайн фронт.md` (разделы 4–5 и аддендумы Н.1–Н.6). Если расходится с основным файлом — основной файл первичен, но здесь ничего не должно расходиться. Всё **обратно совместимо**: существующие эндпоинты (чаты, поисковый стрим, гео-слои) не меняются, только расширяется ответ объекта и добавляется один новый эндпоинт.

---

## 0. TL;DR — что сделать

1. **Расширить** ответ `GET /api/v1/objects/{id}` новыми полями внутри `lifestyle_analysis` (вердикт, бриф, компромиссы, relaxation, обоснование зоны) и расширить объекты `blocks[]` (`tier`, `verdict_line`, типизированный `data`).
2. **Реализовать 3 hero-`data` payload'а** под ключи `family_routing`, `social_environment`, `view_and_climate`.
3. **Добавить новый SSE-эндпоинт** `POST /api/v1/objects/{id}/ask/stream` — чат-вопросы по конкретному объекту.
4. **Зафиксировать enum'ы** (см. §4) и убедиться, что порядок координат везде `[lng, lat]` (WGS84).

Всё новое в §1–§3 **опционально для рендера** (фронт не падает без поля), но для полноценной демо-карточки нужны все.

## 1. Изменение: `GET /api/v1/objects/{object_id}?chat_id={uuid}`

Это ровно тот объект, что фронт кладёт в паспорт-экран (тип `ObjectPassport`). Совпадение имён/типов полей = нулевой маппинг на фронте.

### 1.1 Новые поля верхнего уровня `lifestyle_analysis`

Добавляются **рядом** с существующими `match_score`, `summary`, `blocks` — ничего не переименовываем.

```jsonc
{
  "match_score": 96,
  "summary": "…длинный текст, как и раньше…",

  "verdict": {                          // hero-вердикт (крупный экран сверху)
    "headline": "Идеально по школе и безопасности — компромисс по свету",
    "confidence": 0.9,                  // 0..1 → чип «уверенность модели»
    "layers_checked": 6                 // «Проверено 6 слоёв города»
  },

  "brief": [                            // критерии из запроса (Linguistic-агент)
    { "label": "Школа 239 ≤ 15 мин пешком", "status": "met" },
    { "label": "Без коммуналок в радиусе",  "status": "met" },
    { "label": "Тишина под окнами",         "status": "compromise" }
  ],
  // status ∈ "met" | "compromise" | "relaxed" | "unknown"

  "blocks": [ /* см. 1.2 */ ],

  "compromises": [                      // «честный детектив»
    { "block_key": "view_and_climate",
      "text": "Утром двор в тени соседнего корпуса, прямое солнце после 14:00." }
  ],
  "relaxation": [                       // что сделал Relaxation Loop (можно [])
    { "text": "В радиусе 15 мин без проспектов идеальных вариантов не было — расширил до 18 мин." }
  ],
  "zone_rationale": "Пересечение 15-мин доступности школы 239 и языкового центра, минус кварталы с нерасселённым фондом."
}
```

`compromises` / `relaxation` могут быть пустыми `[]`, но поля **должны присутствовать** (фронт скрывает секцию при пустоте).

### 1.2 Расширение объектов `blocks[]`

К уже согласованным `key`, `title`, `icon`, `score`, `description`, `metrics` добавляются три поля:

```jsonc
{
  "key": "family_routing",
  "tier": "hero",                       // НОВОЕ: "hero" | "secondary" (нет поля → secondary)
  "title": "Суточный ритм семьи",
  "icon": "route",
  "score": "A",
  "verdict_line": "Все доедут вовремя, ребёнок идёт сам и безопасно.",  // НОВОЕ: короткая фраза
  "description": "…",
  "metrics": { "…": 0 },
  "data": { /* НОВОЕ: типизирован под key — см. §2 */ }
}
```

- **`tier`** — `hero` рендерится главой на весь экран с крупной виз; `secondary` — компактной плиткой.
- **`verdict_line`** — одна фраза-вывод по блоку (опционально).
- **`data`** — для `hero`-блоков обязателен; для `secondary` можно опустить. Неизвестный `key` с `data` фронт покажет как generic (текст+оценка) — не ломается.

### 1.3 Полная форма ответа (для сверки)

```jsonc
{
  "id": "jk-neva-residence",
  "name": "ЖК Neva Residence",
  "address": "Санкт-Петербург, Новоладожская ул., 4",
  "price": 18500000,
  "rooms": 3,
  "area_sqm": 78.5,
  "floor": "7/12",
  "images": ["https://cdn.../img1.jpg"],
  "coordinates": [30.2741, 59.9632],     // [lng, lat]
  "lifestyle_analysis": {
    "match_score": 96,
    "summary": "…",
    "verdict": { "...": "см. 1.1" },
    "brief": [ /* см. 1.1 */ ],
    "blocks": [ /* см. 1.2 + §2 */ ],
    "compromises": [ /* см. 1.1 */ ],
    "relaxation": [ /* см. 1.1 */ ],
    "zone_rationale": "…"
  }
}
```

## 2. Три hero-`data` payload'а (главное «напиши всё»)

Ключи hero-блоков фиксированные: **`family_routing`**, **`social_environment`**, **`view_and_climate`**.

### 2.1 `family_routing` — суточный граф семьи

```jsonc
"data": {
  "home": [30.3625, 59.949],           // [lng,lat] — общая точка старта
  "members": [
    {
      "id": "child", "label": "Сын",
      "legs": [
        {
          "to_label": "Лицей 239", "to_kind": "school",   // to_kind ∈ school|metro|work|park|poi
          "mode": "walk",                                 // mode ∈ walk|scooter|bus|car|metro
          "depart": "08:15", "arrive": "08:26", "minutes": 11,
          "safety": "safe",                               // safety ∈ safe|caution
          "geometry": { "type": "LineString", "coordinates": [[30.3625,59.949],[30.3479,59.9439]] }
        }
      ]
    }
    // + members для мамы/папы со своими legs
  ]
}
```

Состав семьи (`members`) — парсинг из запроса Linguistic-агентом («сын», «жена работает в Сити» и т.д.). `geometry` — реальные пешеходные/авто маршруты по дорогам.

### 2.2 `social_environment` — соцслой / скрытый контингент

```jsonc
"data": {
  "home": [30.3625, 59.949],           // опционально; если нет — фронт берёт дефолтный центр
  "radius_m": 500,
  "scores": {                          // 0..1, где 0 — хорошо, 1 — плохо
    "communal_share": 0.05,            // доля нерасселённого/коммунального фонда
    "bars_density": 0.12,              // плотность баров/алкомаркетов
    "crime_index": 0.18                // нормализованный крим-индекс
  },
  "heat": {                            // FeatureCollection для reveal-хитмапа вокруг дома
    "type": "FeatureCollection",
    "features": [
      { "type": "Feature",
        "properties": { "layer": "communal", "weight": 0.9 },   // layer ∈ communal|bars|crime
        "geometry": { "type": "Polygon", "coordinates": [[[30.370,59.940],[30.377,59.940],[30.377,59.946],[30.370,59.946],[30.370,59.940]]] } }
    ]
  },
  "pois": [                            // опционально — подписанные точки
    { "kind": "bar", "coordinates": [30.361,59.941], "label": "Бар (24ч)" }
  ]
}
```

### 2.3 `view_and_climate` — инсоляция 2.0

```jsonc
"data": {
  "orientation_deg": 225,              // азимут окна (0=С, 90=В, 180=Ю, 270=З)
  "direct_light": { "from": "14:00", "to": "18:00" },
  "sun_hours_by_season": { "winter": 1.5, "spring": 4, "summer": 6, "autumn": 3 },
  "cloudiness_factor": 0.6,            // 0..1 понижающий коэф. (Питер пасмурный)
  "obstructions": [                    // затеняющие соседние здания
    { "azimuth_deg": 120, "elevation_deg": 22, "label": "Соседний корпус" }
  ],
  "view_type": "courtyard_park",       // view_type ∈ courtyard_park|street|water|wall|well
  "db": 35                             // фоновый шум под окном, дБ
}
```

## 3. Новый эндпоинт: чат по объекту (Object Q&A)

Диалог «спросить про эту квартиру» прямо в паспорте. Модель отвечает **grounded** — только по dossier этого объекта (§1–§2) + гео-слоям города (раздел 5 основного файла). **Не смешивается** с поисковым чатом `/chats/{id}/messages/stream`.

- **Метод:** `POST`
- **Эндпоинт:** `/api/v1/objects/{object_id}/ask/stream`
- **Content-Type:** `text/event-stream` (SSE)

### 3.1 Request

```json
{
  "text": "Почему компромисс по свету? Утром вообще солнца не будет?",
  "chat_id": "a1b2c3d4-..."
}
```

`chat_id` обязателен — привязывает вопрос к контексту поиска (критерии/приоритеты пользователя). `object_id` — из пути.

### 3.2 Поток событий (SSE) — те же типы, что у поискового движка

- **`text_token`** — стриминг ответа: `data: {"token": "Утром "}`
- **`error`** — сбой: `data: {"code": "llm_timeout", "message": "…"}` (коды — как в разделе 6 основного файла)
- **`stream_end`** — маркер конца успешного потока: `data: {}` — **обязателен** (без него фронт не закроет `EventSource`)
- **`agent_status`** — опционально, для индикатора «печатает…»

### 3.3 Правила

- **Конкурентность:** пока стрим по `(object_id, chat_id)` не завершён, повторный `POST` → `409` (`stream_in_progress`).
- **Отмена:** по обрыву TCP-соединения останавливать вычисления (без отдельного DELETE).
- **Аутентификация SSE:** та же, что выбрана в разделе 1 основного файла (cookie-session рекомендуется — браузерный `EventSource`/`fetch`-stream и Bearer-хедеры несовместимы).
- **Grounding:** только факты из dossier/гео-слоёв объекта; нет данных → честно «не знаю по этому объекту», без выдумок.

## 4. Зафиксированные enum'ы (согласовать ML ↔ бэк ↔ фронт)

| Поле | Допустимые значения |
|---|---|
| `blocks[].tier` | `hero` \| `secondary` |
| hero `blocks[].key` | `family_routing` \| `social_environment` \| `view_and_climate` |
| `brief[].status` | `met` \| `compromise` \| `relaxed` \| `unknown` |
| `blocks[].icon` | `school` \| `users` \| `sun` \| `volume` \| `leaf` \| `hospital` \| `route` |
| `blocks[].score` | `A+`…`D` (см. раздел 4 основного файла) |
| `family_routing … to_kind` | `school` \| `metro` \| `work` \| `park` \| `poi` |
| `family_routing … mode` | `walk` \| `scooter` \| `bus` \| `car` \| `metro` |
| `family_routing … safety` | `safe` \| `caution` |
| `social_environment.heat … layer` | `communal` \| `bars` \| `crime` |
| `view_and_climate.view_type` | `courtyard_park` \| `street` \| `water` \| `wall` \| `well` |
| гео-слои `layers` (раздел 5) | `communal` \| `noise` \| `schools` \| `bars` \| `ecology` \| `parks` |

**Координаты везде `[lng, lat]`, WGS84 (EPSG:4326), без трансформаций на фронте.**

## 5. Интеграция (как фронт подключится)

Фронт написан на data-seam: весь доступ к бэку идёт через тонкий клиент `lib/api/`, компоненты бэк напрямую не трогают. Для перехода с мока на прод бэку достаточно:

1. Поднять эндпоинты: разделы 2–5 основного файла + §1 и §3 отсюда.
2. Выставить на фронте два env-флага:
   - `NEXT_PUBLIC_USE_MOCK=false`
   - `NEXT_PUBLIC_API_BASE=https://<backend-host>/api/v1`
3. Настроить cookie-session (см. §3.3) — работает одинаково для REST и SSE.

UI-код при этом **не меняется**.
