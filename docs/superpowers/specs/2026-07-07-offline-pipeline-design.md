## 1. Контекст и ключевой принцип

Строим ML-часть ИИ-агента по недвижимости для Москвы. **Своя модель не обучается** — ценность в оркестрации агентов и гибридном поиске (RAG) поверх заранее собранной базы. Всё, что похоже на «ML» (эмбеддинги, reranker, парсинг запроса), — это инференс готовых моделей.

Система делится на две фазы:
- **OFFLINE (batch):** сбор данных, гео-обогащение, генерация эмбеддингов, укладка в БД. Вся вычислительная «тяжесть» здесь.
- **ONLINE (real-time):** оркестрация, гибридный retrieval, reranking, объяснение выдачи.

**Эта спека покрывает только OFFLINE-фазу.** Online-фаза проектируется отдельной итерацией.

### Целевой регион
**Москва.** Определяет OSM-экстракт, гео-слои (шум ТТК/ЗСД/трамвайные пути), выбор датасета.

### Зона ответственности и границы
Я (ML/Data) владею всем, что про данные и модели — от скрапинга до финального промпта. **Не трогаю** (зона беков): прод-сервер БД, API-слой/FastAPI-обёртку деплоя, транспорт (REST/gRPC), клиент к ORS/Valhalla. Для разработки поднимаю локальный `docker-compose` (Postgres+PostGIS+pgvector), который беки заменят на своё.

## 2. Источники данных

Датасета и инфраструктуры на старте нет — стартуем с нуля. Два канала данных:

1. **Bootstrap-датасет (Kaggle):** [Russia Real Estate 2018-2021 (mrdaniilak)](https://www.kaggle.com/datasets/mrdaniilak/russia-real-estate-20182021) — ~540k объектов, источник cian.ru. Есть `geo_lat`/`geo_lon`, цена, площадь, этаж/этажность, комнаты, тип объекта. Фильтруем по региону Москва. Это структурный слепок базы.
2. **Свежий слой (скрапер):** скрапер Циана/Домклика → живые объявления **с текстовыми описаниями**. Работает в инкрементальном цикле (upsert по external_id).

### Находка: у Kaggle-датасета нет текстовых описаний
Bootstrap-датасет — числовой, без свободного текста. А гибридный поиск (dense+sparse) держится на тексте («тихий двор», «школа рядом»). Решение:
- **Скрапер даёт настоящие описания** — основной источник живого текста.
- **Для «немых» строк (Kaggle) синтезируем каноничный документ** из структурных + гео-обогащённых полей (text-ification), например:
  `"2-комн, 54 м², 7/12 этаж, окна ЮЗ, школа №239 в 12 мин пешком, баров в 500 м нет, тихо"`.

Так даже bootstrap-строки семантически ищутся, а скрапленные обогащаются ещё и живым текстом. Text-ification — стандартный приём подготовки структурных данных под RAG, и идеальная почва для sparse-канала (точные термины).

## 3. Архитектура и оркестрация

**Выбранный подход:** плоские Python-модули + тонкий CLI + cron (не Prefect/Dagster/Airflow). Спека прямо предписывает `cron`; тяжёлый оркестратор — избыточная инфра на хакатон (YAGNI). Беки легко подхватят простые скрипты.

### Стек
- Python 3.12, `uv` для зависимостей и venv.
- Postgres 16 + PostGIS + pgvector (локальный `docker-compose` для разработки).
- Эмбеддинги: **BAAI/bge-m3** через `FlagEmbedding`/`sentence-transformers` — dense (1024) + sparse из одной модели, self-host, Apache 2.0, силён на кириллице. Апгрейд по качеству (позже, если узко) — Qwen3-Embedding. **Смена embedding-модели = полный реиндекс базы**, поэтому фиксируется на старте.
- OSM: `pyosmium` (диффы) + `osmnx`/Overpass для первичной выгрузки POI.
- Гео: `geopandas` + PostGIS-запросы (`ST_DWithin`, `ST_Distance`).
- Геокодинг: Nominatim (бесплатный, с лимитами) или DaData/Yandex (если будет ключ) — только для строк без координат.

### Структура модулей
Каждый модуль — одна ответственность, тестируется независимо.
```
habitus/
  config.py            # env, пути, константы (город=Москва, radii=500m, dim=1024 …)
  db/
    schema.sql         # listings, poi, индексы (HNSW/GiST/B-tree)
    migrations/        # инкрементальные миграции
  ingest/
    kaggle_loader.py   # скачать датасет + загрузить в raw_listings
    cian_scraper.py    # скрапер Циана/Домклика → raw_listings (с описаниями)
  clean/
    normalize.py       # дедупликация, нормализация адресов, отсев мусора
    geocode.py         # адрес→координаты (только где нет geo), ретраи+кэш
  geo/
    osm_extract.py     # выгрузка POI Москвы (школы/бары/алкомаркеты/парки/дороги)
    enrich.py          # PostGIS: пешая доступность, bar_density_500m, окна/инсоляция, шум
  embed/
    document.py        # структура+гео → каноничный текст (text-ification)
    encode.py          # BGE-M3 dense(1024)+sparse → колонки в БД
  update/
    incremental.py     # OSM-диффы, upsert, каскадный пересчёт ST_DWithin
  cli.py               # `habitus offline run` / `habitus update` — оркестрация шагов
```

### Поток данных
```
kaggle_loader + cian_scraper  →  raw_listings
        ↓ clean.normalize + clean.geocode
   listings (чистые поля + координаты)
        ↓ geo.osm_extract → poi ;  geo.enrich (+POI-производные колонки)
   listings обогащены (bar_density_500m, noise, window_orientation …)
        ↓ embed.document → каноничный текст ;  embed.encode → dense+sparse
   listings полностью готовы к retrieval
        ↓ update.incremental (по cron)
   диффы OSM → upsert POI → каскадный пересчёт затронутых listings
```

## 4. Схема БД (ядро)

Одно хранилище — фильтры, гео и оба типа семантики в одной базе.

**`listings`:**
- Идентификация: `id`, `external_id` (upsert-ключ), `source` (kaggle/cian/domclick), `is_active bool`.
- Структурные поля: `price`, `area`, `kitchen_area`, `rooms`, `level`, `levels`, `building_type`, `object_type`.
- Гео: `geom geometry(Point, 4326)`.
- Гео-производные (offline-считанные): `walk_min_school`, `walk_min_metro`, `walk_min_park`, `bar_density_500m`, `window_orientation` (массив сторон света), `insolation_rough`, `noise_level`.
- Текст/семантика: `description` (nullable — из скрапера), `doc_text` (каноничный текст под эмбеддинг), `embedding vector(1024)`, `sparse_embedding sparsevec`, `content_hash` (для инкрементального реиндекса).
- Freshness: `created_at`, `updated_at`.

**`poi`:** гео-объекты (школы, бары, алкомаркеты, парки, дороги/пути) — `id`, `osm_id`, `kind`, `name`, `rating` (для школ), `geom`, `updated_at`.

**Индексы:**
- HNSW на `embedding` (dense retrieval).
- HNSW на `sparse_embedding` (при необходимости) или через RRF-запрос.
- GiST на `geom` (гео-фильтры `ST_DWithin`).
- B-tree на `price` и other структурных фильтрах.

## 5. Гео-обогащение (PostGIS)

Заранее считаем то, что дорого считать на лету, отдельными колонками — тогда online-запрос это фильтр по готовому полю, а не расчёт:
- Пешая доступность до школ/метро/парков (изохроны-прокси; точные изохроны ORS — на online через беков).
- Плотность баров и алкомаркетов в радиусе 500 м (`ST_DWithin`).
- Ориентация окон по сторонам света + грубая инсоляция (азимут против высотности соседних зданий).
- Шумовые слои (близость ТТК/ЗСД/трамвайных путей).

---

## 6. Инкрементальный цикл (freshness)

База — живой слой города, не разовый слепок. По cron:
- **Гео-объекты:** OSM replication diffs через `pyosmium` — забираем только изменения. Новый бар → upsert в `poi`.
- **Объявления:** повторный прогон источников, upsert по `external_id`; пропавшие → `is_active = false` (история остаётся).
- **Каскадный пересчёт:** новый POI меняет производные колонки квартир вокруг. `ST_DWithin(geom, new_poi, 500)` находит затронутые listings (десятки, не вся база) → пересчитываем только их `bar_density_500m`, шум и т.п.
- **Эмбеддинги только по изменившимся текстам:** `content_hash` описания в колонке; текст не менялся → вектор валиден, не трогаем. Реиндекс не нужен.
- `updated_at` у каждой сущности → система честно показывает «данные актуальны на …».

## 7. Обработка ошибок и надёжность
- Геокодинг: ретраи с backoff + кэш (адрес→координаты), уважение лимитов Nominatim.
- Эмбеддинг: батчинг, `normalize_embeddings=True`, пересчёт только по изменившемуся `content_hash`.
- Скрапер: вежливые задержки, чекпоинты (resume), robots/анти-бот аккуратность, идемпотентность.
- Всё идемпотентно: upsert по `external_id`, повторный прогон безопасен.

---

## 8. Тестирование
- `pytest` на каждый модуль на маленьких фикстурах (10–20 строк).
- PostGIS-запросы — на тестовой БД в docker (реальный PostGIS, не мок).
- Golden-фикстура для text-ification (`embed/document.py`): вход-строка → ожидаемый каноничный текст.
- Проверка идемпотентности upsert (двойной прогон → тот же результат).
- Smoke-тест полного offline-прогона на подвыборке (~1000 строк).

**В этой итерации:** весь offline-пайплайн — ingest (Kaggle + скрапер), clean, geo enrich, embed, инкрементальный update, схема БД, локальный docker, тесты.
