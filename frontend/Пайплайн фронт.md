# 📑 API Specification: Client-Facing Layer (Front ↔ Back)

Документ описывает контракты эндпоинтов, DTO и архитектуру взаимодействия для клиентской части **AI Urban Intelligence Agent**.

> Пункты, помеченные ⚠️, — это решения, которые нужно принять и зафиксировать всей командой **до** старта разработки, а не по ходу дела: их дороже всего переигрывать посреди работы.

---

## 🧭 1. Архитектурный паттерн и протоколы

1. **HTTP REST** — управление сессиями чатов, история, статические данные объектов, тяжёлые гео-слои (GeoJSON).
2. **SSE (Server-Sent Events)** — стриминг поисковых запросов: шаги оркестратора («мысли ИИ») и токены текста.

_Почему SSE, а не WebSocket:_ нужен однонаправленный стриминг сервер → клиент. SSE проще в реализации, работает поверх HTTP, не требует хэндшейка сокетов.

### ⚠️ Аутентификация — решить сейчас

Браузерный `EventSource` **не умеет** отправлять кастомные HTTP-хедеры. Если аутентификация идёт через `Authorization: Bearer`, SSE-эндпоинт сломается на ровном месте. Варианты:

- **Cookie-based session** (рекомендуется) — токен в httpOnly cookie, работает одинаково для REST и SSE, ничего лишнего на фронте.
- Либо токен как query-параметр в SSE-запросе: `/messages/stream?token=...` (менее безопасно, но проще на хакатон).

Все REST-эндпоинты ниже подразумевают, что пользователь аутентифицирован (сессия/токен), и `chat_id` в базе привязан к конкретному `user_id`. Без этого `GET /chats` будет отдавать чаты всех пользователей подряд.

### Формат ошибок (единый для всех REST-эндпоинтов)

```json
{
  "error": {
    "code": "chat_not_found",
    "message": "Чат с указанным ID не найден"
  }
}
```

Используется во всех `4xx`/`5xx` ответах ниже — единообразно, чтобы фронт писал один обработчик ошибок, а не по одному на эндпоинт.

## 🗂️ 2. Эндпоинты управления чатами (Chat Management)

### 2.1 Создание новой сессии чата

- **Метод:** `POST`
- **Эндпоинт:** `/api/v1/chats`

**Request:**

```json
{
  "city": "spb",
  "title": "Новый поиск квартиры"
}
```

`city`: `"spb" | "msk"` — обязательное поле. `title`: опционально, дефолт — `"Новый поиск квартиры"`.

**Response `201 Created`:**

```json
{
  "chat_id": "a1b2c3d4-e5f6-7a8b-9c0d-1e2f3a4b5c6d",
  "city": "spb",
  "title": "Новый поиск квартиры",
  "created_at": "2026-07-08T12:00:00Z"
}
```

### 2.2 Получение списка чатов пользователя

- **Метод:** `GET`
- **Эндпоинт:** `/api/v1/chats?limit=20&offset=0`

**Response `200 OK`:**

```json
{
  "chats": [
    {
      "chat_id": "a1b2c3d4-e5f6-7a8b-9c0d-1e2f3a4b5c6d",
      "city": "spb",
      "title": "Поиск около 239 лицея без коммуналок",
      "updated_at": "2026-07-08T12:40:00Z"
    }
  ],
  "total": 1
}
```

`limit`/`offset` — опциональны, для хакатона можно не пагинировать реально, но поле `total` пусть отдаётся сразу (легче добавить пагинацию на фронте позже).

### 2.3 Переименование чата

- **Метод:** `PATCH`
- **Эндпоинт:** `/api/v1/chats/{chat_id}`

**Request:**

```json
{ "title": "Квартира у Сколково" }
```

**Response `200 OK`:** обновлённый объект чата (как в 2.1).

> Если `title` генерируется автоматически LLM-ом после первого сообщения пользователя — см. SSE-событие `chat_renamed` в разделе 3, чтобы сайдбар обновился в реальном времени.

### 2.4 Удаление чата

- **Метод:** `DELETE`
- **Эндпоинт:** `/api/v1/chats/{chat_id}`
- **Response:** `204 No Content`
- **Response `404 Not Found`:** если чат не найден (формат ошибки — см. раздел 1).

### 2.5 Получение истории сообщений чата

- **Метод:** `GET`
- **Эндпоинт:** `/api/v1/chats/{chat_id}/messages?limit=50&offset=0`

**Response `200 OK`:**

```json
{
  "messages": [
    {
      "message_id": "m1",
      "role": "user",
      "text": "Ищу квартиру в СПб, чтобы ребенок ходил в математическую школу пешком.",
      "created_at": "2026-07-08T12:35:00Z"
    },
    {
      "message_id": "m2",
      "role": "assistant",
      "text": "Я подобрал для вас варианты...",
      "created_at": "2026-07-08T12:36:00Z",
      "meta": {
        "suggested_object_ids": ["jk-neva-residence", "jk-imperial"]
      }
    }
  ],
  "total": 2
}
```

## 🧠 3. Главный поисковый ИИ-движок (Core AI Search Engine)

- **Метод:** `POST`
- **Эндпоинт:** `/api/v1/chats/{chat_id}/messages/stream`
- **Content-Type:** `text/event-stream`

### ⚠️ Правило конкурентных запросов — решить сейчас

Пока стрим для чата не завершился (`stream_end` не пришёл), повторный `POST` в этот же `chat_id` должен возвращать `409 Conflict`. Фронт блокирует инпут на время стрима — но бэк обязан отклонять параллельные вызовы, а не потому что «фронт же не даст», а как гарантия на уровне API.

### Request:

```json
{
  "text": "Ищу евро-трешку. Сын должен ходить в 239 лицей пешком не больше 15 минут, без переходов через проспекты. Важно отсутствие коммуналок в радиусе и чтобы окна во двор."
}
```

### Поток событий (SSE)

**`agent_status`** — статус работы агентов (для лоадеров на фронте):

```
event: agent_status
data: {"agent": "linguistic", "status": "processing", "message": "Выделяю сущности: лицей 239, фильтр 'коммуналки', <15 мин пешком"}
```

```
event: agent_status
data: {"agent": "geo", "status": "processing", "message": "Построение 15-минутных пешеходных маршрутов вокруг лицея..."}
```

```
event: agent_status
data: {"agent": "context", "status": "processing", "message": "Сканирование структуры жилого фонда..."}
```

Значения `agent`: `"linguistic" | "geo" | "context" | "orchestrator"`. Значения `status`: `"processing" | "done" | "relaxation_triggered"`.

**`agent_status` (relaxation)** — если сработал relaxation loop:

```
event: agent_status
data: {"agent": "orchestrator", "status": "relaxation_triggered", "message": "В радиусе 15 минут без проспектов нет идеальных вариантов. Расширяю до 20 минут."}
```

**`text_token`** — стриминг текстового саммари:

```
event: text_token
data: {"token": "Я "}

event: text_token
data: {"token": "нашел "}
```

**`chat_renamed`** — если после первого сообщения LLM сгенерировал название чата:

```
event: chat_renamed
data: {"chat_id": "a1b2c3d4-...", "title": "Поиск около 239 лицея без коммуналок"}
```

**`final_result`** — финальный датасет:

```json
event: final_result
data: {
  "suggested_areas_geojson": {
    "type": "FeatureCollection",
    "features": [
      {
        "type": "Feature",
        "properties": { "area_type": "recommended_zone" },
        "geometry": { "type": "Polygon", "coordinates": [[[30.34, 59.94], [30.35, 59.94], [30.35, 59.95], [30.34, 59.94]]] }
      }
    ]
  },
  "objects": [
    {
      "id": "jk-neva-residence",
      "name": "ЖК Neva Residence",
      "cover_image": "https://cdn.../cover1.jpg",
      "match_score": 96,
      "coordinates": [30.2741, 59.9632],
      "price_from": 18500000,
      "rooms": 3,
      "area_sqm": 78.5,
      "floor": "7/12",
      "tags": ["11 минут до школы", "0% коммуналок вокруг", "Окна во двор-парк"]
    }
  ]
}
```

**`error`** — при сбое на любом шаге пайплайна:

```
event: error
data: {"code": "llm_timeout", "message": "Не удалось получить ответ от ИИ, попробуйте ещё раз"}
```

Возможные `code`: `llm_timeout`, `llm_unavailable`, `geo_service_error`, `db_error`, `internal_error`. При получении `error` фронт останавливает лоадеры и показывает сообщение — стрим на этом завершается (`stream_end` после `error` не отправляется).

**`stream_end`** — явный маркер конца потока (успешного):

```
event: stream_end
data: {}
```

Фронт использует это событие, чтобы закрыть `EventSource` и разблокировать инпут. **Обязательно** к реализации — без него фронт не может отличить «пауза между событиями» от «стрим завершён».

### ⚠️ Важно про поля `final_result.objects[]`

- Поле называется **`match_score`** (то же имя, что и в `/objects/{id}`) — не заводите отдельное `match_percentage`, чтобы не было двух названий для одного значения.
- **Координаты — строго `[lng, lat]`** (стандартный порядок GeoJSON), а не `[lat, lng]`. Перепутанный порядок — самый частый источник багов с картой (точки уезжают не в то полушарие).
- В карточке объекта нужны `cover_image`, `rooms`, `area_sqm`, `floor` — без фото и базовых параметров карточка в списке результатов бесполезна для UI.

### Обработка пустых результатов

Если `Relaxation Loop` не нашёл ничего подходящего даже после расширения критериев:

```json
{
  "suggested_areas_geojson": { "...": "зона поиска сохраняется" },
  "objects": []
}
```

Фронт в этом случае показывает: _«Мы выделили зону поиска, но подходящих квартир в ней сейчас нет»_ — используя `suggested_areas_geojson` для отрисовки зоны на карте.

### Отмена запроса

Если пользователь закрывает `EventSource` до `stream_end` (ушёл со страницы/передумал) — сервер должен по обрыву TCP-соединения останавливать вычисления пайплайна, а не докручивать его впустую. Явный `DELETE`-эндпоинт для отмены не нужен на хакатон — достаточно детектить закрытие соединения на бэке.

## 🏠 4. Детализация объекта: ИИ-Паспорт Лайфстайла (Lifestyle Passport)

- **Метод:** `GET`
- **Эндпоинт:** `/api/v1/objects/{object_id}?chat_id={uuid}`

`chat_id` — **обязателен**, чтобы `lifestyle_analysis` генерировался под контекст конкретного диалога.

### ⚠️ Схема `blocks` — динамическая, не хардкодить набор ключей

В примере ниже 3 блока (`logistics`, `social_environment`, `view_and_climate`), но в других сценариях (например, «пара Сколково↔Сити») набор блоков другой — экология, инфраструктура для ребёнка и т.д. **Фронт должен рендерить `blocks` как массив/словарь generic-карточек** (`title` + `score` + `description`), а не захардкоженные 3 иконки под конкретные ключи. Бэку — не фиксировать состав ключей объекта `blocks`, а отдавать столько блоков, сколько релевантно запросу.

Рекомендация: сделать `blocks` массивом, а не объектом с фиксированными ключами — так фронту гарантированно не придётся ничего доопределять при появлении нового типа блока:

```json
"blocks": [
  {
    "key": "logistics",
    "title": "Логистика и Школы",
    "icon": "school",
    "score": "A+",
    "description": "До Лицея 239 ровно 11 минут пешком..."
  },
  {
    "key": "social_environment",
    "title": "Окружение и Социальный слой",
    "icon": "users",
    "score": "A",
    "description": "Доля нерасселенного старого фонда..."
  }
]
```

`icon` — опциональный ключ иконки из зафиксированного набора (см. ниже), чтобы фронт не гадал, какую картинку подставить под новый тип блока.

**Набор `icon` (согласовать и зафиксировать с ML):** `school`, `users`, `sun`, `volume`, `leaf`, `hospital`, `route` — расширяется по мере необходимости, но должен быть конечным списком, известным фронту заранее.

### Шкала `score` (зафиксировать)

`"A+" | "A" | "A-" | "B+" | "B" | "B-" | "C+" | "C" | "C-" | "D"` — используется фронтом для цветовой индикации (зелёный/жёлтый/красный). Без фиксированного enum фронт не сможет заранее замапить цвета.

### Response `200 OK`:

```json
{
  "id": "jk-neva-residence",
  "name": "ЖК Neva Residence",
  "address": "Санкт-Петербург, Новоладожская ул., 4",
  "price": 18500000,
  "rooms": 3,
  "area_sqm": 78.5,
  "floor": "7/12",
  "images": ["https://cdn.../img1.jpg", "https://cdn.../img2.jpg"],
  "coordinates": [30.2741, 59.9632],
  "lifestyle_analysis": {
    "match_score": 96,
    "summary": "Этот ЖК идеально подходит под ваши критерии безопасности ребенка и близости к обучению, однако имеет компромисс по инсоляции.",
    "blocks": [
      {
        "key": "logistics",
        "title": "Логистика и Школы",
        "icon": "school",
        "score": "A+",
        "description": "До Лицея 239 ровно 11 минут пешком. Маршрут полностью безопасен: проходит через регулируемый пешеходный переход и тихий переулок. Крупные проспекты пересекать не нужно."
      },
      {
        "key": "social_environment",
        "title": "Окружение и Социальный слой",
        "icon": "users",
        "score": "A",
        "description": "Доля нерасселенного старого фонда и коммунальных квартир в радиусе 500 метров составляет 0%. Вокруг новые семейные ЖК бизнес-класса."
      },
      {
        "key": "view_and_climate",
        "title": "Взгляд из окна и свет",
        "icon": "sun",
        "score": "B-",
        "description": "Окна выбранного корпуса выходят на Юго-Запад во внутренний двор-парк. Прямое солнце с 14:00 до 18:00. Уровень шума минимальный (35 дБ)."
      }
    ]
  }
}
```

**Response `404 Not Found`:** объект не найден (единый формат ошибки).

> **Заметка для бэка:** кэшируйте статические поля объекта (`images`, `address`, `price`), но пересчитывайте `lifestyle_analysis` динамически на основе контекста `chat_id`.

## 🗺️ 5. Интерактивные гео-слои для карты (Geo Layers)

- **Метод:** `GET`
- **Эндпоинт:** `/api/v1/geo/layers?city=spb&layers=communal,noise,schools`

### Допустимые значения `layers` (зафиксированный enum)

`communal` (плотность коммунального фонда), `noise` (шумовые магистрали), `schools` (точки школ), `bars` (плотность баров/алкомаркетов), `ecology` (роза ветров / удалённость от промзон), `parks` (точки парков), `metro` (точки станций метро). Список закрытый — если бэк добавляет новый слой, он появляется здесь же, а не как сюрприз на фронте.

### Response `200 OK`:

```json
{
  "city": "spb",
  "layers": {
    "communal": {
      "type": "FeatureCollection",
      "features": [
        {
          "type": "Feature",
          "properties": { "weight": 0.9, "description": "Высокая концентрация коммунального фонда" },
          "geometry": { "type": "Polygon", "coordinates": [ [30.30, 59.93], [30.31, 59.93], [30.31, 59.94] ] }
        }
      ]
    },
    "noise": {
      "type": "FeatureCollection",
      "features": [
        {
          "type": "Feature",
          "properties": { "db_level": 75, "source": "ЗСД / Магистраль" },
          "geometry": { "type": "LineString", "coordinates": [ [30.32, 59.95], [30.33, 59.95] ] }
        }
      ]
    },
    "schools": {
      "type": "FeatureCollection",
      "features": [
        {
          "type": "Feature",
          "properties": { "name": "Президентский ФМЛ №239", "rating": "Top-1 Математика" },
          "geometry": { "type": "Point", "coordinates": [30.3484, 59.9441] }
        }
      ]
    }
  }
}
```

Если запрошенный `layers` не входит в enum — бэк должен молча игнорировать неизвестный слой (не возвращать ошибку), чтобы фронт мог смело запрашивать все известные ему слои без риска сломать запрос.

Все координаты — строго **WGS 84 (EPSG:4326)**, порядок `[lng, lat]`, без трансформаций на фронте.

## 📋 6. Сводная таблица кодов ошибок

|HTTP код|`error.code`|Когда|
|---|---|---|
|`400`|`validation_error`|Невалидный payload запроса|
|`401`|`unauthorized`|Нет / истёк токен сессии|
|`404`|`chat_not_found`|Чат не найден|
|`404`|`object_not_found`|Объект недвижимости не найден|
|`409`|`stream_in_progress`|Повторный запрос в чат, где уже идёт стрим|
|`500`|`internal_error`|Непредвиденная ошибка сервера|

SSE-специфичные коды ошибок (`llm_timeout`, `llm_unavailable`, `geo_service_error`, `db_error`) — см. раздел 3.

## ✅ Чек-лист для бэкендеров перед началом работы

1. Выбрать способ передачи токена в SSE (cookie vs query-param) — см. раздел 1.
2. Зафиксировать конечный список `icon` для lifestyle-блоков — согласовать с ML-частью.
3. Подтвердить: `blocks` — массив (рекомендовано) или объект с фиксированными ключами.
4. Реализовать `error` и `stream_end` события в SSE — без них фронт не может корректно завершать/обрабатывать сбои стрима.
5. Проверить порядок координат `[lng, lat]` во всех местах, включая `final_result.objects[].coordinates`.

# 🆕 ADDENDUM (2026-07-13): Карточка-«Досье детектива» — что добавить

> Это **дополнение** к разделу 4 (`GET /api/v1/objects/{id}`). Всё, что было выше, остаётся как есть и обратно совместимо. Ниже — только **новые поля**, которые нужно добавить в `lifestyle_analysis`, чтобы фронт собрал кинематографичную карточку-паспорт в формате «досье расследования» (hero-вердикт → бриф → главы с hero-виз → прочие улики → честный вывод).
>
> Приоритет для бэка: сначала три `hero`-блока с их `data` (это центр «вау»), затем честный футер (`compromises`/`relaxation`/`zone_rationale`), затем `verdict`/`brief`.

## Н.1 Новые поля верхнего уровня `lifestyle_analysis`

Добавляются **рядом** с существующими `match_score`, `summary`, `blocks` — ничего не переименовываем.

```jsonc
{
  "match_score": 96,
  "summary": "…длинный текст, как и раньше…",

  "verdict": {                          // ① hero-вердикт (крупный экран сверху)
    "headline": "Идеально по школе и безопасности — компромисс по свету",
    "confidence": 0.9,                  // 0..1 → чип «уверенность модели»
    "layers_checked": 6                 // «Проверено 6 слоёв города»
  },

  "brief": [                            // ② бриф расследования — критерии из Linguistic-агента
    { "label": "Школа 239 ≤ 15 мин пешком", "status": "met" },
    { "label": "Без коммуналок в радиусе",  "status": "met" },
    { "label": "Тишина под окнами",         "status": "compromise" }
  ],
  // status (зафиксированный enum): "met" | "compromise" | "relaxed" | "unknown"

  "blocks": [ /* см. Н.2 — добавлены поля tier / verdict_line / data */ ],

  "compromises": [                      // ⑤ «честный детектив» — явные компромиссы
    { "block_key": "view_and_climate",
      "text": "Утром двор в тени соседнего корпуса, прямое солнце после 14:00." }
  ],
  "relaxation": [                       // что сделал Relaxation Loop (можно [] )
    { "text": "В радиусе 15 мин без проспектов идеальных вариантов не было — расширил до 18 мин." }
  ],
  "zone_rationale": "Почему именно эта зона: пересечение 15-мин доступности школы 239 и языкового центра, минус кварталы с нерасселённым фондом."
}
```

Все новые поля **опциональны для рендера**: если поля нет — фронт просто не рисует соответствующую секцию (карточка не ломается). Но для полноценного «досье» на защите нужны все.

## Н.2 Расширение объекта `blocks[]`

К уже согласованным `key`, `title`, `icon`, `score`, `description`, `metrics` добавляем:

```jsonc
{
  "key": "family_routing",
  "tier": "hero",                       // "hero" | "secondary" — НОВОЕ
  "title": "Суточный ритм семьи",
  "icon": "route",
  "score": "A",
  "verdict_line": "Все доедут вовремя, ребёнок идёт сам и безопасно.",  // НОВОЕ: короткая фраза-улика
  "description": "…",
  "metrics": { "…": 0 },
  "data": { /* типизирован под key — см. Н.3 */ }                        // НОВОЕ
}
```

- **`tier`** — как фронт раскладывает блок: `hero` → отдельная глава почти на весь экран с крупной виз; `secondary` → компактная плитка в сетке «прочих улик». Если поле отсутствует — считать `secondary`.
- **`verdict_line`** — одна короткая фраза-вывод по блоку (для закреплённой колонки главы). Опционально.
- **`data`** — типизированный payload под конкретную виз. Для `hero`-блоков обязателен; для `secondary` можно опустить (тогда текст + оценка). Неизвестный `key` с `data` → generic-фолбэк на фронте (текст + оценка), запрос не ломается.

## Н.5 Визуальные ноты для фронта (как рендерятся hero-главы)

Единый «приборный» язык (как в текущей `InsolationViz`), один приглушённый акцент, без неона/пурпура/чистого чёрного, `font-mono` для чисел, `min-h-[100dvh]` для глав, анимации только `transform/opacity`, обязательный `prefers-reduced-motion` (reduced → сразу финальный кадр без проигрыша).

**Каркас главы:** асимметрия `lg:grid-cols-[22rem_1fr]` — слева липкая колонка-улика (иконка, заголовок, оценка, `verdict_line`, метрики моно), справа крупная виз. Разделители `border-t`, без лишних карточек.

- **`family_routing`** → живой maplibre + Gantt-лента дня (дорожка на члена семьи, сегменты по `mode`, тайминги), скраббер 06:00→23:00 двигает маркеры всех персон синхронно; hover сегмента ↔ подсветка маршрута на карте; `safety:"caution"` — пунктирное янтарное подчёркивание. Персоны — 3 десатурированных тона.
- **`social_environment`** → мини-карта с кольцом `radius_m`, сканирующий свип проявляет хит-полигоны (`communal`/`bars`/`crime`), сегментный переключатель слоёв с opacity-кроссфейдом; три `scores` как риск-метры 0..1 (emerald→deep-rose, число моно, цвет не единственный индикатор).
- **`view_and_climate`** → расширение `InsolationViz`: затенённые клинья на дуге по `obstructions`, сезонный переключатель (пружинно меняет дугу и `direct_light`), пасмурный оверлей по `cloudiness_factor`, бейдж `view_type`.

## Н.6 НОВЫЙ ЭНДПОИНТ: чат по конкретному объекту (Object Q&A)

Диалог «спросить про эту квартиру» прямо в паспорте объекта. Модель отвечает, опираясь на dossier этого объекта (раздел 4 + Н.1–Н.3) и гео-слои города (раздел 5). Отдельный поток, **не смешивается** с поисковым чатом (`/chats/{chat_id}/messages/stream`).

- **Метод:** `POST`
- **Эндпоинт:** `/api/v1/objects/{object_id}/ask/stream`
- **Content-Type:** `text/event-stream` (SSE)

### Request

```json
{
  "text": "Почему компромисс по свету? Утром вообще солнца не будет?",
  "chat_id": "a1b2c3d4-..."
}
```

`chat_id` — обязателен: привязывает вопрос к контексту поиска (те же критерии/приоритеты пользователя), чтобы ответ был персонализирован. `object_id` — из пути.

### Поток событий (SSE)

Переиспользуем уже согласованные типы событий поискового движка (раздел 3), чтобы фронт задействовал тот же обработчик:

- **`text_token`** — стриминг ответа по токенам: `data: {"token": "Утром "}`.
- **`error`** — сбой: `data: {"code": "llm_timeout", "message": "…"}` (коды — как в разделе 3/6).
- **`stream_end`** — явный маркер конца успешного потока: `data: {}`. **Обязателен** (без него фронт не закроет `EventSource` и не разблокирует ввод).

Опционально (если удобно ML): **`agent_status`** для лоадера «печатает…» — тот же формат, что в разделе 3.

### Правила

- Правило конкурентности как в разделе 3: пока стрим по `(object_id, chat_id)` не завершён (`stream_end`), повторный `POST` → `409` (`stream_in_progress`).
- Отмена по обрыву соединения (пользователь ушёл) — как в разделе 3, без отдельного DELETE.
- Аутентификация SSE — та же, что выбрана в разделе 1 (cookie-session рекомендуется).
- Ответы должны быть **grounded**: только факты из dossier/гео-слоёв объекта; если данных нет — честно «не знаю по этому объекту», а не выдумка.

### Чек-лист (доп. к Н.4)

11. Реализовать `POST /objects/{id}/ask/stream` с событиями `text_token` / `error` / `stream_end` (переиспользовать SSE-обвязку поискового движка).
12. Ограничить контекст модели dossier'ом конкретного `object_id` + гео-слоями — не давать ей «фантазировать» за пределами данных объекта.

# 🔌 Как фронт подключит бэк (data-seam)

Фронт написан так, что переключение с мока на реальный бэк — **точечное**, без правки UI-компонентов:

- Весь доступ к данным идёт через тонкий клиент `lib/api/` (напр. `getObjectPassport(objectId, chatId)` и стрим-клиент `askObject(objectId, chatId, text, handlers)`), а не через прямые импорты мока в компонентах.
- Флаг `NEXT_PUBLIC_USE_MOCK` (по умолчанию `true`) выбирает mock- или `fetch`/SSE-реализацию. Для интеграции бэку достаточно: поднять эндпоинты разделов 2–5 + Н.6, выставить `NEXT_PUBLIC_USE_MOCK=false` и указать базовый URL — UI-код не меняется.
- Форма ответа `GET /objects/{id}` (раздел 4 + Н.1–Н.3) — это ровно тот `ObjectPassport`, который потребляет паспорт-экран. Совпадение имён/типов полей = нулевая работа по маппингу на фронте.
